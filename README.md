# وثيقة متطلبات — نظام تشغيل وتوزيع مخبز

**الإصدار:** 1.4
**التاريخ:** 28 نيسان 2026 (ميلادي)
**الغرض:** خط أساس للمتطلبات الوظيفية وغير الوظيفية للنظام قيد البناء.
**مصدر السياق:** سيناريو تشغيلي مستنتج من مناقشات أصحاب المصلحة + **تحليل مباشر لملفات Excel الفعلية** لأسبوع توزيع كامل (15–21 مايو 2025) على خط آنتويرب.

---

## 1. النطاق

### 1.1 ضمن النطاق

- **تطبيق ويب واحد** (المتصفح فقط): يخدم دوري **المدير (Admin)** و**الموزّع (Distributor)** من نفس قاعدة الشيفرة مع صلاحيات حسب الدور.
- **واجهة متجاوبة (Responsive)** حتى يستخدم الموزّعون الهاتف في الميدان عبر **متصفح الموبايل**.
- **حلقة التشغيل الأساسية:** محلات ← منتجات ← طلبات ← تعيين موزّع ← تسليم + تسجيل دفع + مرتجعات ← محاسبة + تقارير.
- **الجغرافيا:** لكل محل عنوان نصي + دبوس خريطة + مدينة/منطقة.


---

## 2. التعاريف

| المصطلح | المعنى |
|--------|--------|
| **المدير (Admin)** | مستخدم بصلاحيات كاملة (مالك المخبز / مدير التشغيل). |
| **الموزّع** | سائق توصيل؛ واجهة مقيّدة؛ يرى طلباته فقط ويُسجِّل الدفع والمرتجعات من الهاتف. |
| **المحل / الزبون** | نقطة بيع تستلم منتجات المخبز؛ لها ملف تعريف + موقع. |
| **الطلب** | طلب منتجات لمحل معيّن بتاريخ تسليم محدد؛ يُعيَّن له سائق عند الإنشاء. |
| **جولة اليوم (RouteRun)** | وحدة جامعة لكل طلبات يوم واحد لسائق واحد — تعادل «ورقة اليوم» في Excel. |
| **الدين (CREDIT)** | الزبون استلم البضاعة ولم يدفع — يُضاف تلقائياً لرصيده المتراكم. |
| **رصيد الدين** | المبلغ الإجمالي المتراكم على المحل من طلبات مُسلَّمة غير مدفوعة. |
| **تسديد الرصيد (SETTLEMENT)** | دفعة يسوي بها الزبون جزءاً من دينه المتراكم أو كله. |
| **الدفع المجزأ** | طلب واحد يُدفع جزء منه نقداً وجزء آخر ديناً — مسموح. |
| **المرتجع** | كمية رُفضت أو لم تُسلَّم — يُسجِّلها الموزّع من الميدان، وتُطرح من قيمة الطلب. |
| **هدية الأسبوع** | أكياس خبز تُمنح لزبائن محددين، تُسجَّل كملاحظة في الطلب. |

## 3. السيناريو التشغيلي

### 3.1 الواقع الحالي

- الطلبات تصل ليلاً عبر واتساب أو قنوات غير رسمية.
- اعتمد التشغيل على **جداول Excel يومية** («ورقة لكل يوم توصيل») بتسعة أعمدة: رقم، اسم الزبون، خبز أبيض، خبز أسمر، الواصل (مبلغ أو حالة الدفع)، الدفعة، الباقي، مواد أخرى، المدينة.
- **طرق الدفع الموثَّقة:** نقداً، دين، بالبنك، دفعة جزئية، تسديد رصيد.
- **المرتجعات** تُسجَّل عند عودة السائق.
- **مصروفات الوقود** تُخصم من المقبوضات للوصول للصافي.
- **رصيد الدين** يتراكم على بعض المحلات لأسابيع وتُسوَّى دورياً.

### 3.2 السلوك المستهدف

1. **البيانات الأساسية:** المدير يُضيف المحلات (مع مدينة + خريطة) والمنتجات والموزّعين مرة واحدة.
2. **إنشاء الطلبات:** المدير يُنشئ الطلب ويُحدِّد السائق المسؤول عنه مباشرةً عند الإنشاء — كل طلب له سائق منذ البداية.
3. **التنفيذ الميداني (الموزّع من هاتفه):**
   - يرى قائمة طلباته المعيّنة مرتّبة حسب ترتيب الجولة.
   - لكل طلب: يرى البنود والكميات + **رصيد دين المحل** + ملاحظة الهدية.
   - يُسلِّم البضاعة ← **يُسجِّل طريقة الدفع يدوياً** (نقد / دين / بنك / دفعة / تسديد) مع المبلغ.
   - يُسجِّل أي **مرتجع** في الميدان إن رُفض جزء من الطلب (الطلب الأصلي يبقى كما هو والكمية المُسلَّمة = الأصلية − المرتجع).
   - يضغط «تم التسليم».
4. **إغلاق اليوم (المدير):** يُدخل مصروف الوقود ← يرى **الصافي = مقبوضات − مصروفات** لكل سائق وإجمالاً.
5. **التقارير:** رصيد كل محل + إجماليات يومية + تقرير لكل سائق على حدة.

## 4. الفاعلون والأهداف

| الفاعل | الأهداف الرئيسية |
|--------|-------------------|
| **المدير** | صيانة الكتالوج والمحلات والمستخدمين؛ إنشاء الطلبات + تعيين السائق؛ إدخال مصروفات الوقود؛ مراجعة التقارير وأرصدة الديون. |
| **الموزّع** | رؤية طلباته اليومية؛ تسجيل الدفع والمرتجعات من الهاتف؛ تحديث حالة التسليم. |

## 5. المتطلبات الوظيفية

### المرحلة أ — الأساس

**FR-A1 — إدارة المستخدمين والأدوار**

- إنشاء وتعديل وتعطيل المستخدمين.
- أدوار: **مدير** و**موزّع** — لا يجوز للموزّع الوصول للوحة الإدارة.

**FR-A2 — إدارة المحل**

- CRUD المحل: الاسم، الاتصال، البريد (اختياري)، نوع المحل، عنوان نصي حر.
- **المدينة / المنطقة:** حقل إلزامي (دندرموند، سنت كلاس، برخم... إلخ).
- **موقع الخريطة:** خط عرض/طول إلزامي.

**FR-A3 — كتالوج المنتجات**

- CRUD المنتج: الاسم، SKU، الوحدة، سعر الوحدة، حالة التفعيل.
- سجل تاريخي لتغييرات السعر (من / متى / ماذا تغيّر).

---

### المرحلة ب — الطلبات

**FR-B1 — إنشاء الطلب**

- الطلب مرتبط بمحل واحد + تاريخ تسليم + بنود (مع لقطة سعر).
- **السائق يُحدَّد عند إنشاء الطلب** — لا خطوة تعيين منفصلة.
- حقل ملاحظات يستوعب هدايا الأسبوع والتعليمات الخاصة.
- حالات الطلب: `draft` → `confirmed` → `ready_for_distribution` → `out_for_delivery` → `delivered` / `cancelled`.

**FR-B2 — قائمة العمل اليومية**

- المدير يصفّي طلبات يوم معيّن مع عرض: المحل، الكميات، السائق، رصيد دين المحل.

**FR-B3 — سجل أحداث الطلب**

- تغييرات الحالة والتعديلات مسجَّلة بالفاعل والوقت.

---

### المرحلة ج — التوزيع

**FR-C1 — تعيين السائق عند إنشاء الطلب**

- كل طلب يُعيَّن لسائق واحد مباشرةً من فورم الإنشاء.
- ممكن عدة سائقين في اليوم نفسه — كل سائق يرى طلباته فقط.

**FR-C2 — قائمة عمل الموزّع**

- الموزّع يرى طلباته اليومية فقط — مُرتَّبة حسب ترتيب الجولة.
- كل طلب يُظهر: اسم المحل + المدينة + البنود + **رصيد دين المحل** + ملاحظات.

**FR-C3 — جولة اليوم (RouteRun)**

- وحدة جامعة لطلبات يوم + سائق + سيارة + كميات محمَّلة + كمية احتياطية.
- الكمية الاحتياطية يُدخلها المدير أو السائق — لا فرق.
- تُولَّد منها **ملخص نهاية الجولة**: مقبوضات، مرتجعات، مصروفات، صافي.

**FR-C4 — المركبات**

- كيانات مركبة اختيارية تربط جولة اليوم بسيارة محددة.

---

### المرحلة د — التنفيذ الميداني

**FR-D1 — تحديث الحالة + تسجيل الدفع (الموزّع)**

- الموزّع من هاتفه لكل طلب:
  1. يُسجِّل طريقة الدفع + المبلغ يدوياً قبل الضغط على «تم التسليم».
  2. الدفع المجزأ مسموح: مبلغ نقدي + مبلغ دين في نفس الطلب.
  3. يُسجِّل المرتجع إن وُجد (كمية + نوع المنتج).
  4. يضغط «تم التسليم» — تتغير الحالة تلقائياً.

**FR-D2 — المرتجعات الميدانية**

- الموزّع يُسجِّل المرتجع من الميدان لحظة الرفض.
- الطلب الأصلي **يبقى كما هو** — قيمة التسليم الفعلية = قيمة الطلب − قيمة المرتجع.
- المرتجع يؤثر على رصيد الدين: لو الطلب ديناً فالمرتجع يُنقِّص المبلغ المستحق.

---

### المرحلة هـ — المدفوعات والمحاسبة

**FR-E1 — أنواع الدفع**

الموزّع يُسجِّل الدفع من الهاتف، والمدير يُسجِّله من اللوحة — كلاهما مسموح.

| النوع | الكود | المعنى |
|-------|-------|--------|
| نقداً | `CASH` | مبلغ مُحصَّل فورياً |
| بالبنك | `BANK_TRANSFER` | تحويل إلكتروني + مرجع |
| دين | `CREDIT` | استلم ولم يدفع — يُضاف للرصيد |
| دفعة جزئية | `INSTALLMENT` | مبلغ على حساب الدين القديم |
| تسديد رصيد | `SETTLEMENT` | سداد الدين المتراكم |

**FR-E2 — رصيد الدين (Running Balance)**

- محسوب ديناميكياً — لا يُخزَّن في عمود منفصل.
- **الصيغة:**
  ```
  رصيد_الدين(محل) =
    SUM( قيمة ما تم تسليمه الفعلي )
      WHERE status = 'delivered' AND paymentMethod IN ('CREDIT', 'INSTALLMENT')
    −
    SUM( مبلغ الدفعة )
      WHERE method IN ('CASH', 'BANK_TRANSFER', 'INSTALLMENT', 'SETTLEMENT')
  ```
- يظهر في: صفحة المحل + قائمة الطلبات + واجهة الموزّع (قراءة فقط).
- **لا حد أقصى للدين** — تنبيه مرئي فقط بدون وقف التسليم.

**FR-E3 — الهدايا الأسبوعية**

- تُسجَّل كملاحظة في حقل `notes` على الطلب — لا بند طلب مستقل.
- تُذكر في ملخص الجولة ضمن مواد أخرى.

**FR-E4 — مصروفات الجولة**

- المدير يُدخل مصروف الوقود (ديزل وغيره) على مستوى الجولة.
- **الصافي = مجموع المقبوضات − مجموع المصروفات**.

**FR-E5 — التقارير**

- إجماليات يومية: مقبوضات، مصروفات، صافي.
- تقرير **لكل سائق على حدة**: ما وزّعه، ما جمعه، صافيه.
- تقرير **إجمالي اليوم** لكل السائقين.
- أرصدة ديون المحلات مرتّبة تنازلياً.

---

### الاستقبال الخارجي — جاهزية مستقبلية

**FR-F1 — واتساب / استيراد خارجي**

- غير مطلوب في الإصدار الأول — الإدخال يدوي.
- التصميم يسمح لاحقاً بإضافة API استيراد دون إعادة كتابة الجداول الأساسية.

## 6. المتطلبات غير الوظيفية

- **NFR-1** الواجهة: عربية RTL أساساً؛ اللاتينية مسموحة في حقول العنوان؛ متجاوبة للموزّع على الهاتف.
- **NFR-2** العملة: **اليورو (EUR)** دائماً.
- **NFR-3** استضافة مع HTTPS؛ بيئة تجهيز منفصلة مُنصَح بها.
- **NFR-4** مصادقة + rate limiting + أقل صلاحيات ممكنة.
- **NFR-5** تدقيق على الطلبات والمدفوعات والأسعار.

## 7. الصلاحيات (RBAC)

| العملية | ADMIN | DISTRIBUTOR |
|---------|-------|-------------|
| إدارة المستخدمين | نعم | لا |
| CRUD المحلات والمنتجات | نعم | لا |
| إنشاء الطلبات + تعيين السائق | نعم | لا |
| قائمة الطلبات الكاملة | نعم | لا |
| إنشاء جولة اليوم + مصروفات الوقود | نعم | لا |
| رؤية التقارير الكاملة | نعم | لا |
| رؤية طلباته المعيّنة فقط | لا | نعم |
| رؤية رصيد دين المحل (قراءة) | نعم | نعم |
| تسجيل الدفع (نقد / دين / بنك...) | نعم | **نعم** |
| تسجيل المرتجعات من الميدان | نعم | **نعم** |
| تحديث حالة التسليم | نعم | نعم (طلباته فقط) |

## 8. سيناريوهات طرف لطرف

**السيناريو 1 — يوم عمل كامل (المدير)**

```
تسجيل الدخول
  → Dashboard: طلبات اليوم + إجمالي الديون
  → إنشاء الطلبات (كل طلب: محل + بنود + سائق)
  → تحويل الحالة إلى "جاهز للتوزيع"
  → إنشاء جولة اليوم (كميات محمَّلة + احتياطي)
  → متابعة التسليمات طوال اليوم
  → إدخال مصروف الوقود
  → مراجعة الملخص: مقبوضات − مصروفات = الصافي
  → تقرير الديون: المحلات مرتّبة حسب الرصيد
```

**السيناريو 2 — يوم عمل كامل (الموزّع)**

```
تسجيل الدخول من الهاتف
  → قائمة طلبات اليوم (مرتّبة بترتيب الجولة)
  → لكل طلب:
      يفتح التفاصيل → يرى البنود + رصيد دين المحل + الملاحظات
      يضغط الخريطة → يتنقل للمحل
      عند الوصول:
        ← يُسجِّل الدفع (نوع + مبلغ)
        ← يُسجِّل مرتجع إن وُجد
        ← يضغط «تم التسليم»
  → يكمل باقي الطلبات
  → يعود للمستودع بالمرتجعات
```

**السيناريو 3 — تسوية دين متراكم**

```
الموزّع عند التسليم:
  يرى رصيد دين المحل: 630€
  الزبون يدفع 630€ نقداً
  الموزّع يُسجِّل: SETTLEMENT — 630€
  رصيد الدين يصبح: 0€
```

**السيناريو 4 — دفع مجزأ**

```
طلب قيمته 91€
الزبون يدفع 50€ نقداً + 41€ ديناً
الموزّع يُسجِّل:
  CASH — 50€ (على هذا الطلب)
  CREDIT — 41€ (على هذا الطلب)
رصيد الدين يرتفع بـ 41€
```

**السيناريو 5 — مرتجع ميداني**

```
طلب: 5 أكياس × 13€ = 65€ (حالة: دين)
الزبون رفض 2 أكياس
الموزّع يُسجِّل مرتجع: 2 أكياس خبز أبيض
القيمة الفعلية المُسلَّمة: 3 × 13 = 39€
رصيد الدين يرتفع بـ 39€ فقط (لا 65€)
```

**السيناريو 6 — يومان سائقان**

```
يوم الأحد:
  محمد → 30 طلباً في منطقة أنتورب + سنت كلاس
  شعث → 20 طلباً في منطقة ترن هاوت + مول
كل سائق يرى طلباته فقط
المدير يرى تقرير لكل سائق + تقرير إجمالي
```

## 9. القرارات المحسومة

| السؤال | القرار |
|--------|--------|
| رصيد الدين يُحسَب من؟ | الطلبات المُسلَّمة (status=delivered) فقط |
| الهدايا: بند مستقل؟ | لا — ملاحظة في حقل notes |
| الموزّع يرى رصيد الدين؟ | نعم — قراءة فقط، لمساعدته على التحصيل |
| حد الدين الأقصى؟ | لا يوجد — تنبيه مرئي فقط |
| VAT؟ | لا تُحتسب إطلاقاً |
| من يُسجِّل الدفع؟ | الموزّع من الهاتف (أساساً) + المدير من اللوحة |
| الدفع المجزأ؟ | مسموح — دفعتان مرتبطتان بنفس الطلب |
| طريقة الدفع متى تُحدَّد؟ | عند التسليم فقط — لا عند إنشاء الطلب |
| عدد السائقين في اليوم؟ | متعدد — كل سائق جولة مستقلة |
| الكمية الاحتياطية من يُقررها؟ | المدير أو السائق — لا فرق |
| المرتجع يؤثر على قيمة الطلب؟ | نعم — قيمة التسليم = أصلية − مرتجع |
| تقرير المدير؟ | إجمالي اليوم + تقرير لكل سائق على حدة |
| طباعة أو PDF؟ | لا — غير مطلوب في هذه المرحلة |

## 10. صيغة حساب رصيد الدين

```
رصيد_الدين(محل) =
  SUM( قيمة ما تم تسليمه الفعلي )
    WHERE status = 'delivered'
      AND paymentMethod IN ('CREDIT', 'INSTALLMENT')
  −
  SUM( مبلغ الدفعة )
    WHERE method IN ('CASH', 'BANK_TRANSFER', 'INSTALLMENT', 'SETTLEMENT')
```

**ملاحظة:** قيمة ما تم تسليمه الفعلي = (كمية الطلب − كمية المرتجع) × سعر الوحدة
